## D. 面试官一定会问的问题 + 强答法

### D1. 技术深度（讲清你就赢一半）

**Q：多模型路由依据是什么？为什么不全走云端？**
> 路由看任务类型（`AiTaskKeywordService` 按关键词分 CODE/DESIGN/ANALYSIS/DATA/TEXT）+ 本地健康状态 + 配额。简单任务（摘要/分类/改写）走本地省成本，复杂任务走云端保质量，本地不可用云端兜底。全云端实现简单但成本不可控，且简单任务没必要用最强模型。**收益我用压测两组对比算，不写死比例。**

**Q：流式 fallback 怎么保证不重复输出？**
> 关键是**只在首 token 前 fallback**。`streamWithLocalFallback` 用 `Flux.defer` + `onErrorResume`，一旦开始往下游 emit 内容就不再切 provider，否则用户会看到两段重复开头。

**Q：计费为什么失败/超限/缺价也要记账？**
> 因为成本系统不能有黑洞。`UsageEventService` 对 SUCCESS/FAILED/BLOCKED_BY_QUOTA/BLOCKED_BY_PRICING 都落流水——失败也消耗了上游调用或被拦截，审计和排查都需要。流式拿不到 usage 时我用字符数兜底估算，避免流式调用在成本上"凭空消失"。

**Q：RAG 召回怎么保证质量？RRF 和作用域兜底为什么这么设计？**
> 关键词召回准但漏，向量召回全但飘，RRF（倒数排名融合）让两路互补且不依赖各自分数量纲。作用域兜底是因为用户常常没指定范围：先在附件里找，命中不足再扩到当前对话、整个会话，避免"明明传了文件却搜不到"。（**做完 P1-3 后**：我有 recall@k 对比数据证明融合优于单路。）

**Q：技能系统的 SSRF 和 CLI 危险命令怎么防的？**
> MCP 端点调用前校验目标地址（禁内网/回环），见 `McpSkillProviderSsrfTest`；CLI 走 `SandboxedCommandRunner` 危险命令门 + `DangerLevel` 分级；所有工具调用过 `AuditingToolCallback` 落审计；身份不让模型碰，由后端 ToolContext 注入。

**Q：三层记忆具体怎么存、怎么压缩？**
> 一层：Redis 按 token 的滑动窗口存近期对话；二层：超长对话触发 `ChatSummaryService` 摘要压缩；二层还有跨会话用户画像聚合。token 全程用 JTokkit 精确计，不用字符数估。

### D2. 弱点问题（提前准备，诚实但有补救）

**Q：你这鉴权靠前端传 `X-User-Id`，不是能随便越权吗？**
> （**P0-1 做完前的诚实答法**）"这是我当前最该补的点，原型阶段用轻量会话简化了鉴权，但我把取身份抽象成了 `CurrentUserService` 接口专门为替换留了缝。" →（**做完后**）"已经换成 JWT 过滤链，身份从 SecurityContext 来，业务层零改动。"
> **别狡辩，承认 + 给出已规划/已完成的方案，反而显成熟。**

**Q：压测过吗？瓶颈在哪？QPS 多少？**
> 做完 P1-2 再有底气答。没做之前别吹数字，可以说"我做了功能完备性，性能压测是我下一步在补的，初步看瓶颈会在向量检索和云端 provider 时延"。

**Q：这项目是不是 AI 写的 / 一个人怎么写这么多？**
> **正面接**："我大量用了 AI 辅助编码，这是 2026 年的正常生产力。但架构是我设计的，每个取舍我都能讲——比如为什么流式只在首帧前 fallback、为什么计费要落失败流水、为什么 RAG 要作用域兜底。" 然后**主动让他挑一个模块深聊**。能 defend 就赢，AI 写不写不重要，懂不懂才重要。

**Q：高可用 / 并发上去会怎样？**
> 诚实讲现状（单体、未做水平扩展验证）+ 讲思路（无状态化、Redis 限流已有、DB 连接池、向量检索缓存）。**别声称"生产级高可用"除非做了故障演练。**

## E. 把"AI 痕迹"变成加分项的三件事

1. **做完 P1-2 压测**——一份真实性能报告是"我亲手挣的"，AI 代不了。
2. **一次真实故障复盘**——比如 Ollama 挂了触发回退、Milvus 连接超时，写成 `docs/postmortem/*.md`，面试拿出来。
3. **一份带取舍的设计文档**——为什么选 Milvus 不选 pgvector、为什么 RRF 不用加权和，写清 trade-off。这三样东西，比再加 5000 行代码更能证明你。
